▌ fraud monitoring dashboards

Stability, performance, and feature analysis for transaction fraud models

Three companion dashboards designed around the failure modes that matter in production fraud modeling — model performance, feature health, and the stability/calibration disciplines that catch attacks the headline AUC misses. Demo data is synthetic but parameterized to a realistic 0.4–0.9% fraud-rate problem with 84 features across 630k transactions.

10 weeks · 70 days of transactions 630k transactions 84 engineered features 2 trained models (GBM champion, LR challenger) Regime shift planted at week 8

Each hub is a standalone HTML file — open it directly, no server required. The dashboards share the same dark-sidebar layout and cross-link to each other. The "demo" variants are populated by real metrics computed from a synthetic transaction dataset with an entity-driven generator (accounts → cards → merchants → devices → IPs, with proper time-windowed velocity features and multiple fraud archetypes).

invariant features (device, behavioral, structural) tactical features (current attack surface) compositional (aggregations / embeddings)

The story across the three hubs (demo data)
Headline AUC holds GBM AUC 0.891 train → 0.783 OOT looks defensible at the model-review level.
Calibration breaks ECE grows ~5× across splits (0.0008 train → 0.0042 recent). Threshold-based policy is now miscalibrated against the new population.
Every cohort degrades All five channels show 9–13 point AUC drops train→OOT — uniform degradation, symptomatic of regime change rather than one bad slice.
Adversarial validation flags drift A train-vs-OOT classifier hits AUC 0.65 from features alone — the feature space is meaningfully different from training.
Tactical features lead the drift Top adversarial discriminators are the velocity/MCC/declines — exactly the features expected to track an evolving attack surface.
Concept > distribution drift Most features show subtle distribution shifts but real univariate AUC drops — the relationship between features and the target moved more than the features themselves.
Source & reproduction

The Python builder under builder/ consumes a scored DataFrame + trained model and emits the three reports. Run python demo_fraud.py to regenerate everything end-to-end (~33s on a hot dataset cache). Templates use a single __DATA_JSON__ placeholder so any payload that matches the schema can drive the visualizations.

The dataset generator (builder/dataset.py) is entity-driven: persistent account/card/merchant/device/IP tables, properly time-windowed velocity computed via two-pointer sweep per card, and a planted BIN-cluster attack starting on day 49 of the 70-day window. Fraud rates calibrate to 0.42% pre-shift and 0.83% post-shift.
